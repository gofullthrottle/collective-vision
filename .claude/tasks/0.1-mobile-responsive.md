# Epic 0.1: Mobile Responsive Design

## Methodology Guidance
**SPECTRA Phase**: Foundation/Hardening
**Approach**: Audit-first, mobile-first CSS implementation
**Tools**: Browser DevTools, iOS Simulator, Android Emulator

## Wave Context
**Wave**: 0 - Foundation Hardening
**Priority**: Critical (blocks all other waves)
**Dependencies**: None
**Estimated Duration**: 4 hours

## Quality Requirements
- Widget renders correctly at 320px, 375px, 768px, 1024px viewports
- No horizontal scrolling on mobile devices
- Touch targets minimum 44x44px (Apple HIG / Material Design)
- Widget fully usable on iPad and functional on mobile

---

## Tasks

### 0.1.1 Audit Current CSS (1h)
**Objective**: Document all responsive issues and establish breakpoint strategy

**Steps**:
1. Review widget CSS (`WIDGET_JS` in worker.ts) for responsive issues
2. Review admin dashboard CSS (if exists)
3. Document breakpoint strategy:
   - Mobile: 320-768px
   - Tablet: 768-1024px
   - Desktop: 1024px+
4. Create audit document listing all components needing updates

**Output**: `docs/responsive-audit.md` with issue list and breakpoint strategy

**Acceptance Criteria**:
- [ ] All current responsive issues documented
- [ ] Breakpoint strategy clearly defined
- [ ] Priority order for fixes established

---

### 0.1.2 Implement Widget Responsive Styles (1.5h)
**Objective**: Make widget fully responsive using mobile-first approach

**Steps**:
1. Add CSS custom properties for responsive values:
   ```css
   :root {
     --cv-spacing-base: 16px;
     --cv-font-size-base: 14px;
     --cv-touch-target: 44px;
   }
   ```
2. Implement mobile-first breakpoints in widget CSS
3. Fix feedback list layout on small screens (stack cards vertically)
4. Fix form inputs for touch devices (larger tap targets, proper spacing)
5. Test on iOS Safari viewport (handle address bar, safe areas)

**Testing Checkpoints**:
- [ ] 320px viewport (iPhone SE)
- [ ] 375px viewport (iPhone 12/13)
- [ ] 768px viewport (iPad portrait)
- [ ] 1024px viewport (iPad landscape)

**Acceptance Criteria**:
- [ ] Widget renders correctly at all target viewports
- [ ] No horizontal scrolling on any mobile device
- [ ] All touch targets meet 44x44px minimum
- [ ] Form inputs have adequate padding for touch

---

### 0.1.3 Implement Admin Dashboard Responsive (1h)
**Objective**: Make admin dashboard functional on tablet and usable on mobile

**Steps**:
1. Implement responsive sidebar:
   - Collapsible hamburger menu on mobile (<768px)
   - Slide-out overlay navigation
2. Make data tables responsive:
   - Horizontal scroll for wide tables on mobile
   - OR card view alternative for key data
3. Responsive charts and stat cards (stack on mobile)
4. Mobile navigation pattern with bottom nav or hamburger

**Acceptance Criteria**:
- [ ] Admin fully usable on iPad (no feature loss)
- [ ] Admin functional on mobile (core features accessible)
- [ ] Navigation intuitive on all screen sizes
- [ ] Charts readable on tablet and mobile

---

### 0.1.4 Touch Interactions (0.5h)
**Objective**: Ensure all interactions work correctly on touch devices

**Steps**:
1. Add touch-friendly hover states:
   - Replace `:hover` with `:active` for mobile
   - Use media query `@media (hover: hover)` for hover-only styles
2. Implement swipe gestures where appropriate:
   - Swipe to dismiss/archive feedback items
   - Horizontal scroll indicators
3. Test on real iOS and Android devices (or high-fidelity simulators)

**Acceptance Criteria**:
- [ ] No hover-only interactions (all accessible via tap/click)
- [ ] Touch feedback (visual) on all interactive elements
- [ ] Smooth touch scrolling
- [ ] Works on iOS Safari and Chrome Android

---
